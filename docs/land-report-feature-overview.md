# 土地报告生成功能说明

## 1. 功能一句话

本功能用于将结构化的土地估价项目数据，自动生成一份完整的土地估价报告 Markdown 成果，覆盖封面、摘要、估价对象界定、土地估价结果及其使用、附件等核心内容。

它的目标不是“让大模型自由写一篇报告”，而是把：

1. 结构化输入数据
2. 报告规则与模板
3. 大模型生成能力

组合成一条可重复、可调整、可测试的报告生成流水线。

## 2. 业务价值

### 2.1 对业务的价值

- 降低人工撰写土地估价报告的时间成本。
- 将报告格式、章节结构、常用表达统一下来，减少不同人员之间的风格偏差。
- 对于重复性高、口径要求强的内容，可以通过模板和规则保证稳定性。
- 对于需要归纳、概括、提炼的内容，可以引入 LLM 提升生成效率。

### 2.2 对管理的价值

- 能把“个人经验”沉淀成“规则 + 模板 + 测试”。
- 能逐步将不稳定的自由生成内容收敛成更可控的模板驱动内容。
- 每个分部和整份报告都可以单独测试，便于验收和迭代。

## 3. 输入与输出

### 3.1 输入

当前标准输入文件为：

- `inputs/land_input.json`

输入数据以结构化 JSON 为主，典型内容包括：

- 项目信息
- 委托估价方
- 估价对象
- 估价依据
- 附件清单
- 估价师

其中，“估价对象”是最核心的数据块，承载宗地用途、容积率、开发程度、面积、总地价、证号、权利情况等关键信息。

### 3.2 输出

当前测试产物以 Markdown 为主，输出目录为：

- `inputs/_outputs/`

其中完整报告测试会生成：

- `inputs/_outputs/land_report.md`

各分部也支持单独输出，例如：

- `land_cover.md`
- `land_summary.md`
- `land_object_definition.md`
- `land_result_usage.md`
- `land_attachments.md`

## 4. 整体生成流程

系统核心入口是 `WordPipeline`。从输入到输出，整体可以理解为五步：

1. 读取规则集
   选择本次要生成的是完整报告，还是某个单独分部。
2. 加载结构化输入
   读取 `land_input.json`，把 JSON 转成内部元素映射。
3. 计算中间字段
   对模板中需要但输入里没有直接给出的内容，进行提取、转换或调用 LLM 生成。
4. 按模板组装章节
   将各个字段填充到模板中，形成最终 Markdown 文本。
5. 输出结果
   写入 `_outputs` 目录，供人工检查、导出或继续处理。

## 5. 模块拆分

完整土地报告由一个总规则集驱动，也支持拆成多个子规则集单独生成。

### 5.1 完整报告

- `ruleset_land`

作用：

- 生成整份土地估价报告。

### 5.2 封面

- `ruleset_land_cover`

作用：

- 生成报告封面信息。

### 5.3 摘要

- `ruleset_land_summary`

作用：

- 生成项目名称、委托方、估价目的、地价定义、估价结果、签章、结果一览表等摘要内容。

### 5.4 估价对象界定

- `ruleset_land_object_definition`

作用：

- 生成估价对象界定、登记状况、权利状况、利用状况等内容。

### 5.5 土地估价结果及其使用

- `ruleset_land_result_usage`

作用：

- 生成估价依据、方法选择、前提条件、报告使用说明、特殊事项等内容。

### 5.6 附件

- `ruleset_land_attachments`

作用：

- 生成附件目录。

## 6. 生成逻辑的三种方式

当前系统不是单一生成方式，而是三种方式混合：

### 6.1 纯模板填充

适用于：

- 已有固定格式，输入里已有明确字段的内容。

特点：

- 最稳定
- 最容易测试
- 风险最低

### 6.2 提取/转换生成

适用于：

- 输入是结构化数据，但需要做拼接、汇总、格式化的内容。

例如：

- 日期转换
- 结果一览表生成
- 固定句式说明生成

特点：

- 不依赖 LLM
- 可控性强
- 适合逐步替代自由生成

### 6.3 LLM 生成

适用于：

- 需要归纳、概括、整理语言表达的内容。

例如：

- 项目名称
- 估价目的描述
- 地价定义概述
- 土地利用状况等

特点：

- 表达能力强
- 适合复杂归纳
- 但稳定性和一致性依赖 prompt、样例和输入质量

## 7. 当前设计思路

整体设计遵循一个核心原则：

**能模板化的尽量模板化，必须概括归纳的部分再交给 LLM。**

这意味着：

- 报告骨架由规则集和模板控制
- 结构清晰、格式固定的段落尽量走提取/转换
- 真正需要语言理解和综合表达的部分才调用 LLM

这样做的好处是：

- 可维护性更强
- 结果更稳定
- 后续优化更容易定位问题

## 8. 测试方式

项目当前已经支持按分部测试，也支持完整报告测试。

### 8.1 完整报告测试

测试文件：

- `test/test_land_pipeline.py`

作用：

- 调用 `ruleset_land`
- 生成完整土地报告 Markdown
- 输出到 `inputs/_outputs/land_report.md`

### 8.2 分部测试

现有分部测试包括：

- `test/test_land_cover_pipeline.py`
- `test/test_land_summary_pipeline.py`
- `test/test_land_object_definition_pipeline.py`
- `test/test_land_result_usage_pipeline.py`
- `test/test_land_attachments_pipeline.py`

### 8.3 运行方式

示例：

```bash
python -m unittest test.test_land_pipeline -v
```

说明：

- 当前测试依赖真实模型配置。
- 若缺少 `DOUBAO_MODEL_KEY` 和 `DOUBAO_MODEL_NAME`，完整生成测试无法执行。

## 9. 当前边界与风险

### 9.1 输入质量依赖较高

如果 `land_input.json` 中字段缺失、口径不一致、命名变化，可能会影响部分章节生成效果。

### 9.2 LLM 内容仍有波动

虽然模板和示例能约束输出，但纯 LLM 生成的部分仍可能出现：

- 表达不一致
- 细节遗漏
- 个别字段理解偏差

### 9.3 需要持续把高频稳定内容模板化

越是格式固定、数字密集、逐项列举的内容，越适合从 LLM 自由生成迁移到“提取/转换 + 模板”模式。

## 10. 后续优化方向

建议下一阶段重点做三件事：

### 10.1 提升结果段稳定性

将“估价结果”这一类结构高度固定的内容，继续从 LLM 自由生成迁移为模板驱动。

### 10.2 缩小 LLM 责任边界

让 LLM 更多承担：

- 数据概括
- 语句润色
- 归纳表达

而不是承担：

- 数值拼接
- 表格生成
- 固定格式段落编排

### 10.3 建立更明确的验收标准

例如针对每个章节建立：

- 必须出现的字段
- 不允许遗漏的数字
- 固定句式要求
- 回归测试样例

## 11. 给老板的简短结论

这套系统已经不是单纯“用 AI 写报告”，而是在搭建一个“土地估价报告自动生产引擎”：

- 输入是标准化业务数据
- 中间是规则、模板和必要的 AI 能力
- 输出是可测试、可复用、可扩展的完整报告

它的意义不只是提高单次写报告的效率，更重要的是把报告生产能力从“依赖个人经验”，逐步转成“依赖系统能力”。
